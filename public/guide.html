Trajectory Viewer Platform Guide — Rubric Review & Creation
Version 1.0 · April 2026
Important: The Trajectory Viewer is a review and preparation tool. It helps you study trajectories, understand rubrics, and prepare your evaluations. All actual tasking, grading, and submission must be done on the Outlier platform. The Trajectory Viewer does not replace Outlier — it supplements it.

1. What is the Trajectory Viewer?

The Trajectory Viewer is an internal tool built for ACC Data Annotation that lets you:

  • Compare AI agent trajectories side-by-side across multiple models for the same task
  • Review auto-generated rubrics that were created using a multi-model evaluation pipeline
  • Understand task structure — milestones, planned interactions, guardrails, and problem statements
  • Bookmark and annotate specific turns for deeper review

Think of it as your workspace for studying the task before you grade on Outlier.

2. Getting Started

Step 1 — Log In

Navigate to the platform URL and log in with the credentials provided by your admin. Each annotator gets a unique username and password tied to a specific task.

Step 2 — Overview Tab

You'll land on the Overview tab. This shows the task name, problem statement, and a card for each model with key stats: milestone completion, turn count, tool call breakdown, and planned interactions matched.

Step 3 — Read the Task First

Before looking at any trajectories, read the problem statement and expand the milestone table to understand what the agents were asked to do. This context is critical for evaluating rubrics later.

Step 4 — Review Trajectories

Click a model tab or model card to view the full conversation. Each prompt/response pair is collapsible. Tool calls (Bash, Read, Edit, etc.) are color-coded by type. Use the star icon to bookmark turns you want to revisit, and add notes to saved turns.

Step 5 — Study the Rubrics

Scroll to Evaluation Rubrics on the Overview tab. Use the dropdown filters to focus on a specific type or milestone. Each rubric shows its criterion, type, importance, milestone, and rationale.

Step 6 — Go to Outlier

Once you've studied the task, trajectories, and rubrics in the Trajectory Viewer, switch to Outlier to complete your actual grading and submission.

First-time users: The platform has a built-in guided tour. Click the Guide button in the top-right corner at any time to walk through each part of the interface step by step.

4. Understanding Rubrics

Rubrics are the grading criteria used to evaluate each trajectory. They are auto-generated using a multi-step AI pipeline and then reviewed by the admin. Each rubric is binary: PASS or FAIL.

Rubric Fields

FieldDescription
CriterionThe specific behavior being evaluated. Should be clear enough to determine PASS/FAIL from the trajectory alone.
TypeOne of: correctness, code_style, communication, agent_behavior
MilestoneWhich milestone this rubric is tied to (or "All" if it spans the entire task)
ImportanceMUST_FOLLOW (critical), GOOD_TO_HAVE (meaningful signal), or UNIVERSAL (generic, last resort)
is_positivetrue = PASS means the agent DID this. false = PASS means the agent did NOT do this anti-pattern.
RationaleWhy this rubric matters and what it helps distinguish.

Rubric Types Explained

TypeScopeWhat to Look For
correctnessFinal state of the environmentWere milestones achieved? Does the code meet requirements? Do outputs match expectations?
code_styleAgent-generated code onlyReadability, idiomatic patterns, avoidance of redundancy, use of existing config/constants
communicationAgent-to-user messagesClarifying questions, progress updates, explanations, summaries, appropriate conciseness
agent_behaviorFull trajectory of actionsPlanning before coding, reading before editing, reproducing bugs, running tests, recovery from errors

5. The Seven Pillars of Good Rubrics

When reviewing rubrics in the Trajectory Viewer (or writing your own on Outlier), every rubric should satisfy these principles:

  1. Atomic — One behavior per rubric. If you see "and" connecting two checks, it should be split.
  2. Objective — Clear PASS/FAIL with no subjectivity. Avoid "good," "clean," "appropriate" unless precisely defined.
  3. Self-Contained — Evaluable using only the criterion text + trajectory + task definition. No external references needed.
  4. Positively Framed — When is_positive: true, describe what the agent DOES. When false, describe the specific anti-pattern.
  5. Scoped to the Task — Tied to specific milestones. Only evaluate behaviors relevant to the problem statement.
  6. Unique — No two rubrics of the same type should evaluate the same evidence.
  7. Not Overfitting — Valid alternative approaches must also pass. Rubrics must be tool/harness-agnostic.

6. Grading Scale (for Outlier)

After assigning PASS/FAIL for each rubric on every trajectory, you will assign an overall rating on Outlier:

RatingLabelDescription
5PerfectPasses all MUST_FOLLOW and nearly all GOOD_TO_HAVE. No significant failures.
4GoodPasses all MUST_FOLLOW, most GOOD_TO_HAVE. Minor issues only.
3OkayPasses most MUST_FOLLOW, some GOOD_TO_HAVE gaps. May have needed minor hints.
2Low QualityFails some MUST_FOLLOW. Needed significant corrections. Poor agent behavior.
1BadFails multiple MUST_FOLLOW rubrics. Fundamentally flawed approach.

7. Recommended Workflow

  1. Study the task in the Trajectory Viewer — read the problem statement, milestones, and guardrails
  2. Skim all model cards on the Overview tab to get a high-level sense of performance differences
  3. Read each trajectory — focus on milestone boundaries, tool usage patterns, and how the agent handled corrections
  4. Bookmark key turns using the star icon — add notes about what you noticed (good or bad)
  5. Review the rubrics — understand what each criterion is asking and which milestone it targets
  6. Switch to Outlier — complete your PASS/FAIL grading and overall ratings there
  7. Write your rationale on Outlier — reference specific rubrics and turns that drove your scores
Remember: The Trajectory Viewer is your prep workspace. Outlier is where you submit your work. Do not skip the Outlier submission — work done only in the Trajectory Viewer is not recorded.

8. Quick-Reference Checklist

Use this when reviewing rubrics or writing your own on Outlier:

  • Is the criterion atomic? (One behavior only, no "and" bundles)
  • Is it objective? (Clear PASS/FAIL, no subjective words without definition)
  • Is it self-contained? (Evaluable with criterion + trajectory + task definition only)
  • Is it correctly framed? (is_positive: true = describes what agent DOES; false = anti-pattern)
  • Is it scoped to a milestone?
  • Is it unique? (No overlap with other rubrics of the same type)
  • Is it general enough? (Valid alternative approaches would also pass)
  • Is it specific enough? (Bad trajectories would fail)
  • Does it discriminate? (At least one trajectory differs from the others)
  • Is the type correct? (correctness / code_style / communication / agent_behavior)
  • Is the importance correct? (MUST_FOLLOW / GOOD_TO_HAVE / UNIVERSAL)

9. Common Mistakes to Avoid

  1. Vague criteria — "Good formatting," "clean code" without measurable standards
  2. Bundled checks — Multiple behaviors joined by "and." Split them into separate rubrics.
  3. Not self-contained — Requiring re-reading the codebase to evaluate. Include all context in the criterion.
  4. Overfitting to one trajectory — Criteria so specific that only one model's approach passes
  5. Harness-specific language — Referencing specific tool names or commands. Keep rubrics tool-agnostic.
  6. Missing milestone coverage — Every milestone should have at least one rubric
  7. Non-discriminating rubrics — If all trajectories get the same score, the rubric adds no value
  8. Ignoring hint-dependence — Not penalizing models that needed user corrections when others didn't
